MySQL 常用查询优化
大纲指南

面向 IT 程序员的手绘风清单信息图。把索引、SQL、表结构、执行计划、缓存、配置、慢查询与工程实践整理成一张可快速浏览的优化地图。

手绘线条 便利贴布局 低饱和配色 流程化阅读 适合打印/分享

优化流程建议

1发现慢查询：先看慢查询日志，确定高频、耗时和影响范围。
2分析执行计划：用 EXPLAIN 看访问类型、索引命中和扫描行数。
3针对性优化：调整索引、重写 SQL、必要时改表结构或参数。
4验证效果：比对优化前后耗时、扫描行数和资源占用。
5持续监控：保留日志与监控，避免回归与新热点出现。

一、索引优化

先解决“能不能快找到数据”

1.1 索引类型选择

  • B-Tree 索引（最常用）
  • 唯一索引
  • 复合索引
  • 全文索引
  • 覆盖索引
1.2 设计原则
  • 最左前缀原则
  • 高选择性列优先
  • 避免冗余索引
  • 控制索引列长度
1.3 失效场景
  • 函数或表达式
  • 隐式类型转换
  • LIKE 前导通配符
  • OR 条件不当
  • != / <> / IS NULL

二、SQL 语句优化

减少无效读取与多余计算

2.1 查询优化

  • 避免 SELECT *
  • 使用 LIMIT 限制结果集
  • 合理使用 JOIN 代替子查询
  • EXISTS vs IN 的选择
  • UNION vs UNION ALL
2.2 WHERE 优化
  • 过滤条件前置
  • 避免 WHERE 中使用函数
  • 优先走索引列过滤
  • 优化 OR 条件
2.3 排序与分组
  • ORDER BY 利用索引
  • GROUP BY 优化
  • DISTINCT 注意代价

三、表结构优化

从数据设计层面降低查询成本

3.1 数据类型选择

  • 使用最小数据类型
  • 整型优先原则
  • VARCHAR vs CHAR
  • 避免 TEXT / BLOB 滥用

3.2 范式与反范式

  • 合理遵循三范式
  • 适当冗余提升查询效率
  • 分区表使用场景

四、执行计划分析

看懂 EXPLAIN，比盲改更重要

4.1 EXPLAIN 使用

  • type 类型优先级：system > const > eq_ref > ref > range > index > ALL
  • key：是否真正用到了预期索引
  • rows：扫描行数是否过大
  • Extra：关注 Using index、Using filesort 等信息
排查重点
  • 全表扫描是否可避免
  • 回表是否过多
  • 排序/临时表是否过重
实践建议
  • 优化前后都执行一次
  • 结合真实数据量评估
  • 只改最关键瓶颈点

五、查询缓存优化

缓存有效时很快，无效时要果断绕开
  • 查询缓存配置
  • 缓存命中条件
  • 缓存失效场景

阅读提醒

  • 缓存不是万能优化手段
  • 更新频繁的表往往收益有限
  • 更应关注索引与 SQL 本身的质量

六、数据库配置优化

参数调优要贴合业务负载

6.1 关键参数

  • innodb_buffer_pool_size
  • query_cache_size
  • sort_buffer_size
  • tmp_table_size
配置思路
  • 优先解决内存热点
  • 结合排序与临时表使用情况
  • 线上调整要分阶段验证

七、慢查询分析

先量化问题，再决定优化方向

7.1 慢查询日志

  • 开启慢查询日志
  • 设置阈值时间
  • 定期分析慢查询
推荐动作
  • 按次数与耗时双维度排序
  • 合并同类 SQL 模板
  • 优先处理高收益问题

八、其他优化技巧

工程层面的小技巧，往往很实用
8.1 批量操作
  • 批量 INSERT
  • 批量 UPDATE
8.2 连接管理
  • 避免长事务
  • 及时释放连接

8.3 分页优化

  • 延迟关联
  • 记录上次位置
  • 避免大偏移量

程序员速记：排查顺序怎么抓？

建议按照“日志 → 执行计划 → 索引/SQL → 参数/结构 → 回归验证”推进
先看日志
  • 找出最慢、最多、最影响用户体验的 SQL
  • 避免一上来就“凭感觉”改代码
再看 EXPLAIN
  • 判断是否走索引、是否出现大量扫描与排序
  • 确认瓶颈是查找、连接、排序还是回表
最后做针对性变更
  • 优先最小改动、高收益方案
  • 上线后复测并纳入持续监控
常见误区
  • 索引越多越好
  • 缓存一定能救性能
  • 慢就一味加机器
